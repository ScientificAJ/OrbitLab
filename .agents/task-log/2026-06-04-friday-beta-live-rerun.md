## FINDING #1 (real bug, fixed) — Kepler AstroNet ML never scores
- Symptom: Kepler-10 result candidates show ml.label="ml-unavailable" with
  detail "Kepler TensorFlow Docker inference failed: ... SyntaxError: invalid
  syntax" at scripts/predict_kepler_astronet_tf.py:126.
- /models reports kepler_astronet status=ready, so the contract claims a usable
  model while every Kepler inference silently falls back to ml-unavailable.
- Root cause: DockerTensorFlowAstroNetRuntime pins image
  tensorflow/tensorflow:1.5.0-py3 = Python 3.5.2. The helper script used
  f-strings (3.6+), so it dies on first parse INSIDE the container even though
  it is valid on the host (3.12). This means Kepler AstroNet has never produced
  a score in any run on this pinned image.
- Fix: replaced the 3 f-strings in scripts/predict_kepler_astronet_tf.py with
  .format() (py3.5-safe). Verified: parses clean on host py3.12 AND inside the
  pinned TF 1.5.0 container (py3.5.2 -> "PY3.5 SYNTAX OK").
- Follow-up: re-run the Kepler case after the sweep so the report reflects a
  real AstroNet probability instead of the stale ml-unavailable fallback.

## FINDING #2 (suspicious, needs isolated test) — Nigraha TESS prob pinned 0.3
- Nigraha returns raw_ml_probability == 0.3 EXACTLY for every TESS candidate
  across cases 1 & 2 (8 distinct input_tensor_checksums -> still 0.3).
- nigraha_service.predict has no 0.3 default; it returns mean(ensemble sigmoid).
  So the network itself is collapsing to a fixed point.
- Likely driver: imputed_features shows Teff/Radius/logg/Mass/lum/rho are ALL
  imputed (catalog stellar params missing), so scalar inputs are constant
  defaults; combined with saturated conv/dense paths the ensemble can degenerate
  to a constant. Not proven yet.
- Impact: conservative; all affected candidates were rejected anyway, so the
  constant 0.3 (< 0.4 threshold) did not cause a false promotion. But the ML
  surface is not contributing real discrimination on TESS in this run.
- Action: document as a finding; deeper isolation (feed perturbed tensors,
  inspect per-layer activations) is a separate investigation, not in scope of
  "re-run + verify" unless user wants it. NOT changing model code blindly.
